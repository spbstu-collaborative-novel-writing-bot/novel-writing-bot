# Требования к программному проекту

**Наименование проекта:** Collaborative Novel Writing Bot  
**Тип проекта:** Telegram-бот для совместного написания литературных текстов  

---

## 1. Общие сведения

### 1.1. Назначение документа

Настоящий документ содержит требования к программному проекту **Collaborative Novel Writing Bot**. Документ определяет назначение системы, состав пользователей, функциональные требования, нефункциональные требования, требования к технологиям, хранению данных, тестированию, документации и сдаче проекта.

Документ предназначен для участников команды и преподавателя.

### 1.2. Назначение проекта

Проект предназначен для организации совместного написания небольших литературных произведений через Telegram-бота. Пользователь должен иметь возможность создать произведение, добавить главы, пригласить соавторов, просматривать текст и получать текстовые подсказки от языковой модели.

Основная цель проекта -- реализовать небольшой, но завершенный учебный программный продукт, который можно собрать, запустить, проверить и продемонстрировать.

### 1.3. Краткая характеристика системы

Система представляет собой серверное Java-приложение, которое:

1. принимает команды пользователей через Telegram;
2. хранит сведения о пользователях, произведениях, соавторах, главах и истории изменений;
3. проверяет права доступа пользователей к произведениям;
4. отправляет задачи на обработку языковой моделью через RabbitMQ;
5. сохраняет состояние LLM-запросов;
6. предоставляет HTTP-эндпоинты для проверки состояния приложения и административной панели;
7. запускается локально и через Docker Compose.

### 1.4. Границы проекта

В обязательную часть проекта входит:

1. Telegram-бот с текстовыми командами;
2. автоматическая регистрация пользователя по Telegram `chat_id`;
3. роли внутри произведения: владелец и соавтор;
4. создание, просмотр и удаление произведений;
5. добавление и удаление соавторов;
6. создание, просмотр, изменение и удаление глав;
7. Mini App редактор главы с сохранением через HTTP API;
8. хранение истории изменений глав;
9. не менее двух LLM-функций;
10. передача LLM-запросов через RabbitMQ;
11. хранение данных в PostgreSQL;
12. работа с PostgreSQL через Spring JDBC;
13. HTTP-эндпоинт `GET /healthcheck`;
14. административные HTTP-эндпоинты с доступом по административному HTTP-токену или через Telegram Mini App пользователя с ролью `ADMIN`;
15. ручное удаление произведений администратором с уведомлением всех авторов;
16. документирование HTTP-эндпоинтов через Spring REST Docs;
17. сборка исполняемого fat JAR;
18. запуск через Docker Compose;
19. README.md с инструкцией запуска и проверки.

В обязательную часть проекта не входит:

1. регистрация по логину и паролю;
2. полноценный публичный REST API для произведений и глав;
3. одновременное редактирование одной главы несколькими пользователями без optimistic conflict;
4. экспорт в PDF или DOCX;
5. полноценная Git-like система ветвления, слияния и одновременного редактирования текста;
6. расписание напоминаний;
7. промышленная система мониторинга и масштабирования.

## 3. Пользователи и роли

### 3.1. Пользователи Telegram

3.1.1. Пользователи Telegram-бота имеют системную роль `USER` или `ADMIN`; права внутри конкретных произведений задаются отдельно через роли владельца и соавтора.

3.1.2. При первом обращении к боту пользователь должен сохраняться в базе данных по `chat_id`, `username`, `display_name` и системной роли.

3.1.3. Пользователи, чьи `chat_id` указаны в `TELEGRAM_ADMIN_CHAT_IDS`, при регистрации должны получать роль `ADMIN`.

3.1.4. Роль `ADMIN` должна храниться в PostgreSQL и может назначаться или сниматься через административную панель без перезапуска приложения.

3.1.5. Административный доступ к данным и операциям HTTP-панели должен предоставляться по заголовку `X-Admin-Token` или по валидному `X-Telegram-Init-Data` пользователя с ролью `ADMIN`.

### 3.2. Роли внутри произведения

Внутри конкретного произведения должны использоваться следующие роли:

| Роль | Описание |
|---|---|
| `OWNER` | Владелец произведения. Назначается автоматически при создании произведения. |
| `CO_AUTHOR` | Соавтор произведения. Назначается владельцем произведения. |

У одного произведения может быть несколько владельцев. Последнего владельца нельзя удалить.

### 3.3. Матрица прав доступа

| Действие | Владелец | Соавтор | Пользователь без доступа | Администратор API |
|---|:---:|:---:|:---:|:---:|
| Создать произведение | Да | Да | Да | Нет, если не является Telegram-пользователем |
| Просмотреть доступное произведение в Telegram | Да | Да | Нет | Только если имеет доступ как автор |
| Удалить произведение в Telegram | Да | Нет | Нет | Только если является владельцем |
| Добавить главу | Да | Да | Нет | Только если имеет доступ как автор |
| Изменить главу | Да | Да | Нет | Только если имеет доступ как автор |
| Удалить главу | Да | Да | Нет | Только если имеет доступ как автор |
| Просмотреть историю главы | Да | Да | Нет | Только если имеет доступ как автор |
| Добавить соавтора | Да | Нет | Нет | Только если является владельцем |
| Удалить соавтора | Да | Нет | Нет | Только если является владельцем |
| Просмотреть административную статистику | Нет | Нет | Нет | Да |
| Удалить произведение из админ-панели | Нет | Нет | Нет | Да, после уведомления авторов |

## 4. Требования к технологиям

### 4.1. Язык программирования

4.1.1. Проект должен быть реализован на Java 25.

4.1.2. Код должен использовать актуальные возможности Java и не должен быть написан в стиле устаревших версий языка.

### 4.2. Сборка

4.2.1. Для сборки проекта должен использоваться Gradle.

4.2.2. Сборка должна создавать исполняемый fat JAR.

4.2.3. Приложение должно запускаться командой вида:

```bash
java -jar app.jar
```

### 4.3. Spring Framework

4.3.1. Для организации приложения должен использоваться Spring Framework 7.

4.3.2. Использование Spring Boot не допускается.

4.3.3. HTTP-эндпоинты должны быть реализованы с использованием Spring WebFlux.

4.3.4. Конфигурация компонентов Spring должна быть выполнена вручную: через Java-конфигурацию, XML-конфигурацию или их сочетание.

4.3.5. В проекте должны быть выделены компоненты для:

1. обработки Telegram-команд;
2. бизнес-логики произведений и глав;
3. проверки прав доступа;
4. работы с PostgreSQL;
5. отправки и обработки RabbitMQ-сообщений;
6. интеграции с LLM API;
7. HTTP-эндпоинтов;
8. конфигурации приложения.

### 4.4. База данных

4.4.1. Для хранения данных должен использоваться PostgreSQL.

4.4.2. Доступ к базе данных должен быть реализован через Spring JDBC.

4.4.3. PostgreSQL должен запускаться через Docker Compose отдельным сервисом.

4.4.4. После перезапуска контейнера приложения данные не должны пропадать.

### 4.5. Очередь сообщений

4.5.1. Для асинхронной обработки LLM-запросов должен использоваться RabbitMQ.

4.5.2. Приложение должно отправлять в очередь задачи на генерацию текста.

4.5.3. Приложение должно обрабатывать задачи из очереди и сохранять результат или ошибку выполнения.

4.5.4. RabbitMQ должен запускаться через Docker Compose отдельным сервисом.

### 4.6. Telegram API

4.6.1. Приложение должно интегрироваться с Telegram API.

4.6.2. Бот должен принимать текстовые команды от пользователей.

4.6.3. Бот должен отправлять пользователю текстовые ответы.

4.6.4. Допускается использовать long polling. Реализация webhook не является обязательной.

### 4.7. LLM API

4.7.1. Приложение должно интегрироваться минимум с одним LLM API.

4.7.2. В качестве LLM API допускается использовать GigaChat или OpenAI-compatible endpoint.

4.7.3. Ключ LLM API должен передаваться через переменную окружения и не должен храниться в репозитории.

4.7.4. При недоступности LLM API приложение не должно завершаться аварийно.

4.7.5. Для GigaChat приложение должно использовать OAuth-запрос с `RqUID`, `Authorization: Basic ...`, URL-encoded `scope`, затем вызывать chat completions с `Authorization: Bearer <access_token>`.

4.7.6. Для GigaChat должна быть предусмотрена настройка сертификата через `GIGACHAT_CA_CERT_PATH` и флаг проверки SSL `GIGACHAT_VERIFY_SSL`.

4.7.7. В таблице LLM-запросов должна сохраняться короткая техническая причина ошибки без публикации секретов.

### 4.8. Spring REST Docs

4.8.1. Для HTTP-эндпоинтов `GET /healthcheck`, административной панели, Admin JSON API и Mini App API должна быть подготовлена документация с использованием Spring REST Docs или статического раздела HTTP-документации.

4.8.2. Документация должна содержать:

1. путь эндпоинта;
2. HTTP-метод;
3. необходимость авторизации;
4. пример успешного ответа;
5. пример ошибки доступа для административных эндпоинтов.

## 5. Функциональные требования

### 5.1. Регистрация пользователя

5.1.1. При первом выполнении команды `/start` система должна создать пользователя в базе данных.

5.1.2. Для пользователя должны сохраняться:

1. `chat_id`;
2. `username`, если он доступен;
3. отображаемое имя, если оно доступно;
4. дата и время регистрации.

5.1.3. Повторное выполнение команды `/start` не должно создавать дубликат пользователя.

5.1.4. После команды `/start` бот должен отправлять приветственное сообщение и краткий список основных команд.

### 5.2. Справка и обработка неизвестных сообщений

5.2.1. Команда `/help` должна выводить список доступных команд с краткими примерами.

5.2.2. При вводе неизвестной команды бот должен сообщать, что команда не распознана, и предлагать использовать `/help`.

5.2.3. Если пользователь отправляет фото, стикер, голосовое сообщение, видео, файл или другой нетекстовый объект, бот должен отвечать сообщением:

```text
Пожалуйста, используйте команды и кнопки. Введите /help для краткой справки.
```

### 5.3. Управление произведениями

5.3.1. Пользователь должен иметь возможность создать произведение.

Сценарий:

```text
/new
<бот пошагово запрашивает название, описание и жанр>
```

Пример:

```text
/new
Звезды над городом
История о будущем мегаполисе
фантастика
```

5.3.2. После создания произведения пользователь становится его владельцем.

5.3.3. При создании произведения должны сохраняться:

1. идентификатор произведения;
2. название;
3. описание;
4. жанр;
5. `chat_id` владельца;
6. дата создания;
7. дата последнего изменения.

5.3.4. Пользователь должен иметь возможность получить список произведений, к которым он имеет доступ.

Команда:

```text
/novels
```

5.3.5. В списке произведений должны отображаться название, жанр, количество глав, дата обновления и роль пользователя в произведении.

5.3.6. Пользователь должен иметь возможность просмотреть краткую информацию о произведении.

Сценарий:

```text
/novels
<выбрать роман кнопкой>
```

5.3.7. Краткая информация о произведении должна содержать название, описание, жанр, создателя, список владельцев и соавторов, количество глав и дату создания.

5.3.8. Владелец должен иметь возможность удалить произведение.

Сценарий:

```text
/novels
<выбрать роман>
Удалить роман
```

5.3.9. При удалении произведения должны удаляться его главы, связи с авторами и история изменений глав.

5.3.10. Пользователь без доступа к произведению не должен видеть сведения о нем.

### 5.4. Совместная работа

5.4.1. Владелец произведения должен иметь возможность добавить владельца или соавтора по Telegram-тегу `@username`.

Сценарий:

```text
/novels
<выбрать роман>
Авторы
Добавить владельца или Добавить соавтора
```

5.4.2. Добавлять можно только пользователя, который уже запускал бота и существует в базе данных.

5.4.3. Если пользователь с указанным `@username` не найден, бот должен вывести сообщение об ошибке.

5.4.4. Нельзя повторно добавить пользователя, который уже является владельцем или соавтором этого произведения.

5.4.5. Владелец должен иметь возможность удалить участника произведения.

Сценарий:

```text
/novels
<выбрать роман>
Авторы
Удалить участника
```

5.4.6. Последний владелец произведения не может быть удален из списка участников.

5.4.7. Автор произведения должен иметь возможность просмотреть список авторов.

Сценарий:

```text
/novels
<выбрать роман>
Авторы
```

5.4.8. В списке авторов должны отображаться `chat_id`, `username` при наличии, отображаемое имя и роль в произведении.

### 5.5. Работа с главами

5.5.1. Автор произведения должен иметь возможность добавить главу.

Сценарий:

```text
/novels
<выбрать роман>
Добавить главу
```

5.5.2. При добавлении главы должны сохраняться:

1. идентификатор главы;
2. идентификатор произведения;
3. название главы;
4. текст главы;
5. порядковый номер главы;
6. автор создания;
7. автор последнего изменения;
8. дата создания;
9. дата последнего изменения.

5.5.3. Автор произведения должен иметь возможность получить список глав.

Сценарий:

```text
/novels
<выбрать роман>
Главы
```

5.5.4. В списке глав должны отображаться номер главы, название, дата обновления и примерный объем.

5.5.5. Автор произведения должен иметь возможность просмотреть текст конкретной главы.

Сценарий:

```text
/novels
<выбрать роман>
Главы
<выбрать главу>
```

5.5.6. Автор произведения должен иметь возможность просмотреть полный текст произведения.

Сценарий:

```text
/novels
<выбрать роман>
Полный текст .txt
```

5.5.7. Полный текст произведения должен формироваться из глав в порядке их добавления.

5.5.8. Автор произведения должен иметь возможность изменить название и текст главы.

Сценарий:

```text
/novels
<выбрать роман>
Главы
<выбрать главу>
Открыть редактор, Заменить текст, Добавить в конец или Переименовать
```

5.5.9. Перед изменением главы старая версия должна сохраняться в истории изменений.

5.5.10. Автор произведения должен иметь возможность удалить главу.

Сценарий:

```text
/novels
<выбрать роман>
Главы
<выбрать главу>
Удалить главу
```

5.5.11. Автор произведения должен иметь возможность просмотреть историю изменений главы.

Сценарий:

```text
/novels
<выбрать роман>
Главы
<выбрать главу>
История
```

5.5.12. История изменений должна хранить старые версии главы перед изменениями; текущая версия хранится в таблице глав.

5.5.13. В Mini App должен быть доступен постраничный список версий главы, включая текущую версию.

5.5.14. В Mini App должен быть доступен word-level diff выбранной версии с предыдущей.

5.5.15. Пользователь без доступа к произведению не должен иметь возможности просматривать, изменять или удалять его главы.

### 5.6. LLM-функции

5.6.1. Автор произведения должен иметь возможность запросить продолжение главы.

Сценарий:

```text
/novels
<выбрать роман>
Главы
<выбрать главу>
LLM продолжить
```

5.6.2. Автор произведения должен иметь возможность запросить совет по развитию сюжета, персонажей или стиля.

Сценарий:

```text
/novels
<выбрать роман>
Совет LLM
```

5.6.3. Автор произведения должен иметь возможность создать черновик главы по текстовому запросу.

Сценарий:

```text
/novels
<выбрать роман>
Черновик LLM
```

5.6.4. В Web-редакторе главы должны быть доступны LLM-действия: продолжить главу, получить совет и подготовить черновик.

5.6.5. LLM-запрос должен отправляться в RabbitMQ как задача.

5.6.6. После постановки задачи в очередь бот должен сообщить пользователю идентификатор запроса.

Пример ответа:

```text
Запрос принят в обработку. Номер запроса: 15.
```

5.6.7. Пользователь должен иметь возможность проверить статус LLM-запроса.

Команда:

```text
/request_status <request_id>
```

5.6.8. Для LLM-запроса должны поддерживаться статусы:

1. `QUEUED` — запрос поставлен в очередь;
2. `PROCESSING` — запрос обрабатывается;
3. `DONE` — ответ получен;
4. `ERROR` — при обработке возникла ошибка.

5.6.9. При статусе `DONE` бот должен показать результат генерации.

5.6.10. Сгенерированный текст не должен автоматически сохраняться в произведение.

5.6.11. При формировании запроса к LLM приложение должно передавать краткий контекст произведения: название, жанр, описание и при необходимости фрагмент главы.

5.6.12. Чтобы запросы к LLM не были слишком большими, допускается передавать только последние 1000–2000 символов текста главы.

5.6.13. При ошибке LLM API бот должен сохранить статус `ERROR` и вывести понятное сообщение:

```text
Не удалось получить ответ от языковой модели. Попробуйте позже.
```

5.6.14. Если ключ LLM API не задан, LLM-команды должны быть недоступны, но остальные функции бота должны продолжать работать.

### 5.7. HTTP-эндпоинты

5.7.1. Приложение должно предоставлять публичный эндпоинт:

```text
GET /healthcheck
```

5.7.2. Ответ `/healthcheck` должен содержать:

1. статус приложения;
2. название проекта;
3. список авторов проекта;
4. текущую дату и время сервера.

Пример ответа:

```json
{
  "status": "UP",
  "project": "Collaborative Novel Writing Bot",
  "authors": ["Гвоздева Е.", "Крутиков Д.", "Михайлова А.", "Романова А."],
  "time": "2026-04-30T12:00:00"
}
```

5.7.3. Приложение должно предоставлять административную Web-панель:

```text
GET /admin
```

`GET /admin` отдает HTML-оболочку панели. Сами административные данные и операции должны выполняться через защищенные JSON-эндпоинты `/admin/api/*`.

5.7.4. Приложение должно предоставлять административные JSON-эндпоинты:

```text
GET /admin/api/overview
GET /admin/api/users
POST /admin/api/users/{chatId}/role
GET /admin/api/novels
GET /admin/api/llm-requests
POST /admin/api/novels/{novelId}/delete
```

5.7.5. Эндпоинты `/admin/api/*` должны быть доступны при наличии корректного административного токена или валидного `X-Telegram-Init-Data` пользователя с ролью `ADMIN`.

5.7.6. Административный токен должен передаваться в HTTP-заголовке:

```text
X-Admin-Token: <token>
```

Альтернативная авторизация из Telegram Mini App должна передаваться в HTTP-заголовке:

```text
X-Telegram-Init-Data: <Telegram WebApp initData>
```

5.7.7. Ответ `/admin/api/users` должен содержать список зарегистрированных пользователей: `chat_id`, `username`, `display_name`, системную роль, дату регистрации, количество доступных произведений и количество произведений во владении.

5.7.8. Эндпоинт `POST /admin/api/users/{chatId}/role` должен назначать пользователю роль `USER` или `ADMIN`.

5.7.9. Административная статистика должна содержать количество пользователей, администраторов, романов, глав, участников, владельцев, соавторов, объем текста и состояние LLM-запросов.

5.7.10. При отсутствии или неверном значении административного токена, а также при отсутствии или невалидности Telegram Mini App initData, сервер должен возвращать ошибку доступа.

5.7.11. Публичный REST API для управления произведениями и главами не требуется; эти действия выполняются через Telegram и Mini App.

5.7.12. Администратор должен иметь возможность вручную удалить произведение через Web-панель или `POST /admin/api/novels/{novelId}/delete`.

5.7.13. При админском удалении должна передаваться причина удаления длиной от 1 до 1000 символов.

5.7.14. Перед удалением приложение должно отправить всем владельцам и соавторам сообщение с причиной удаления и `.txt` файл с полным текстом произведения.

5.7.15. Если хотя бы одному автору не удалось отправить уведомление или файл, произведение не должно удаляться, а ответ должен содержать `failed_chat_ids`.

5.7.16. Mini App API должен принимать заголовок `X-Telegram-Init-Data`, проверять HMAC-подпись по `TELEGRAM_BOT_TOKEN` и отклонять данные с `auth_date` старше 24 часов.

5.7.17. Mini App API должен предоставлять эндпоинты редактора:

```text
GET /mini/chapter-editor
GET /mini/api/chapters/{novelId}/{chapterId}
PUT /mini/api/chapters/{novelId}/{chapterId}
GET /mini/api/chapters/{novelId}/{chapterId}/history
GET /mini/api/chapters/{novelId}/{chapterId}/versions
GET /mini/api/chapters/{novelId}/{chapterId}/versions/{versionNumber}/diff
POST /mini/api/chapters/{novelId}/{chapterId}/llm
GET /mini/api/llm/{requestId}
```

5.7.18. При некорректных идентификаторах в пути Mini App API должен возвращать JSON-ошибку `400`; при отсутствии или ошибке `X-Telegram-Init-Data` — `403`.

5.7.19. При optimistic conflict сохранения главы Mini App API должен возвращать `409` и актуальную версию главы.

### 5.8. Валидация входных данных

5.8.1. Если пользователь передал недостаточно аргументов, бот должен вывести пример корректного использования команды.

5.8.2. Если идентификатор запроса или данные кнопки переданы в неверном формате, бот должен вывести сообщение об ошибке.

5.8.3. Название произведения не должно быть пустым.

5.8.4. Название главы не должно быть пустым.

5.8.5. Текст главы не должен быть пустым.

5.8.6. Причина админского удаления произведения не должна быть пустой.

5.8.7. Рекомендуемые ограничения длины:

| Поле | Ограничение |
|---|---:|
| Название произведения | до 100 символов |
| Жанр | до 50 символов |
| Описание произведения | до 1000 символов |
| Название главы | до 100 символов |
| Текст главы за одно добавление или изменение | до 200000 символов |
| Вопрос или запрос к LLM | до 1000 символов |

5.8.7. Если текст ответа слишком длинный для одного сообщения Telegram, бот должен отправить сокращенную версию и указать, что текст был сокращен.

## 8. Нефункциональные требования

### 8.1. Производительность

8.1.1. Обычные команды, не обращающиеся к LLM API, должны обрабатываться без заметной задержки для пользователя.

8.1.2. Для LLM-команд допускается более длительное ожидание, так как они выполняются через очередь.

8.1.3. Система должна быть рассчитана на учебную нагрузку:

1. до 20 активных пользователей;
2. до 50 произведений;
3. до 300 глав суммарно;
4. до 100 LLM-запросов.

8.1.4. Реализовывать горизонтальное масштабирование не требуется.

### 8.2. Надежность

8.2.1. Некорректные команды не должны приводить к аварийному завершению приложения.

8.2.2. Ошибки Telegram API, PostgreSQL, RabbitMQ и LLM API должны обрабатываться явно.

8.2.3. При недоступности RabbitMQ LLM-команды могут временно не работать, но остальные команды бота должны продолжать работать, если это технически возможно.

8.2.4. При невозможности сохранить данные бот должен сообщать пользователю, что действие не выполнено.

8.2.5. При запуске без обязательных переменных окружения приложение должно выводить понятное сообщение об ошибке.

### 8.3. Безопасность

8.3.1. Токены Telegram API, LLM API и административный HTTP-токен не должны храниться в репозитории.

8.3.2. Все секреты должны передаваться через переменные окружения или локальный конфигурационный файл, не добавляемый в Git.

8.3.3. Пользователь не должен иметь доступ к произведениям, где он не является владельцем или соавтором.

8.3.4. Только владелец произведения может управлять списком соавторов.

8.3.5. Эндпоинты `/admin/api/*` должны быть защищены административным токеном или валидным Telegram Mini App initData пользователя с ролью `ADMIN`.

8.3.6. В сообщениях об ошибках не должны выводиться токены, ключи и другие служебные секреты.

### 8.4. Удобство использования

8.4.1. Все сообщения Telegram-бота должны быть на русском языке.

8.4.2. Ответы бота должны быть достаточно короткими для чтения в Telegram.

8.4.3. Команда `/help` должна содержать список основных команд и примеры использования.

8.4.4. Идентификаторы произведений, глав и LLM-запросов должны показываться пользователю в списках и ответах, где они нужны для дальнейших команд.

### 8.5. Сопровождаемость

8.5.1. Код должен быть разделен на логические компоненты.

8.5.2. Основная бизнес-логика не должна находиться непосредственно в классе Telegram-бота.

8.5.3. Работа с Telegram, PostgreSQL, RabbitMQ, LLM API и HTTP-эндпоинтами должна быть разделена по разным классам или пакетам.

8.5.4. Названия классов, методов и переменных должны отражать их назначение.

8.5.5. Не допускается оставлять пустые блоки `catch`.

11.1.2. Тесты не должны обращаться к реальному LLM API.

11.1.3. Для проверки LLM-сервиса допускается использовать заглушку.

11.1.4. Для `/healthcheck`, административных эндпоинтов, админского удаления произведений и Mini App API должны быть тесты или документированные сценарии.

## 14. Критерии приемки

Проект считается выполненным, если:

1. приложение собирается в fat JAR;
2. приложение запускается локально;
3. приложение запускается через Docker Compose;
4. в Docker Compose запускаются приложение, PostgreSQL и RabbitMQ;
5. Telegram-бот отвечает на `/start` и `/help`;
6. пользователь может создать произведение;
7. пользователь может просмотреть список своих произведений;
8. пользователь может добавить главу;
9. пользователь может просмотреть главу;
10. пользователь может просмотреть полный текст произведения или его сокращенную версию;
11. владелец может добавить соавтора;
12. владелец может удалить соавтора;
13. пользователь без доступа не может просматривать и изменять чужое произведение;
14. при изменении главы сохраняется история изменений;
15. хотя бы две LLM-команды создают запрос через RabbitMQ;
16. пользователь может проверить статус LLM-запроса;
17. результат LLM-запроса сохраняется и может быть показан пользователю;
18. `GET /healthcheck` доступен без авторизации;
19. административные JSON-эндпоинты доступны только с административным токеном или через Telegram Mini App пользователя с ролью `ADMIN`;
20. администратор может удалить произведение с обязательной причиной после отправки авторам полного текста;
21. данные пользователей, произведений, глав и LLM-запросов сохраняются после перезапуска;
22. HTTP-эндпоинты задокументированы через Spring REST Docs;
23. в проекте есть не менее пяти автоматических тестов;
24. в README.md есть инструкция запуска и примеры команд;
25. подготовлен отдельный файл с пояснениями для защиты проекта;
26. Docker-образ опубликован в Docker Hub;
27. токены и ключи не опубликованы в репозитории.
