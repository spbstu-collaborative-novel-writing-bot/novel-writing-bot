# Требования к программному проекту

**Наименование проекта:** Collaborative Novel Writing Bot  

---

## 1. Общие сведения

### 1.1. Назначение документа

Настоящий документ содержит требования к программному проекту, представляющему собой Telegram-бота для совместного написания небольших литературных произведений. Документ определяет назначение системы, состав пользователей, функциональные и нефункциональные требования, ограничения реализации, требования к исходному коду, сборке, контейнеризации и сдаче проекта.

### 1.2. Назначение проекта

Проект предназначен для организации простой совместной работы над текстом романа через Telegram. Пользователь должен иметь возможность создать роман, добавить главы, пригласить соавторов, просмотреть текст произведения и получить помощь от языковой модели.

Система должна демонстрировать следующие элементы:

1. работу Telegram-бота;
2. хранение пользовательских данных;
3. разграничение прав доступа;
4. взаимодействие с LLM API;
5. наличие HTTP-интерфейса для проверки состояния приложения и административной функции;
6. сборку приложения в fat JAR;
7. запуск приложения в Docker.

Система не должна рассматриваться как полноценный онлайн-редактор документов. Основная цель проекта — реализовать небольшой, но законченный программный продукт, который можно запустить, проверить и продемонстрировать преподавателю.

### 1.3. Краткая характеристика системы

Разрабатываемая система представляет собой серверное приложение, которое принимает сообщения от пользователей через Telegram API, обрабатывает команды, сохраняет данные о пользователях, романах и главах, а также обращается к LLM API для генерации текстовых подсказок.

Взаимодействие с пользователем выполняется в текстовом виде. Основные команды вводятся в Telegram. Для проверки состояния приложения и просмотра списка пользователей используется простой HTTP-интерфейс.

### 1.4. Границы проекта

В обязательную часть проекта входит:

1. Telegram-бот с командами для работы с романами и главами;
2. простая регистрация пользователя по Telegram ID;
3. роли `User` и `Admin`;
4. хранение данных после перезапуска приложения;
5. работа с LLM API минимум по двум сценариям;
6. HTTP-эндпоинт `/healthcheck`;
7. HTTP-эндпоинт `/users` только для администратора;
8. сборка fat JAR;
9. Dockerfile и docker-compose.yml;
10. README.md с инструкцией запуска.

В обязательную часть проекта не входит:

1. веб-интерфейс для редактирования текста;
2. регистрация по логину и паролю;
3. сложная система прав доступа;
4. одновременное редактирование главы несколькими пользователями;
5. экспорт в PDF или DOCX;
6. автоматические напоминания по расписанию;
7. сложный анализ противоречий между главами;
8. промышленная система логирования, мониторинга и масштабирования.

## 2. Термины и сокращения

| Термин | Описание |
|---|---|
| Telegram-бот | Программа, с которой пользователь взаимодействует через сообщения в Telegram. |

## 3. Пользователи и роли

### 3.1. Виды пользователей

В системе должны быть предусмотрены две роли пользователей.

| Роль | Краткое описание |
|---|---|
| `User` | Обычный пользователь Telegram-бота. Может создавать романы, добавлять главы и работать с романами, к которым имеет доступ. |
| `Admin` | Администратор. Имеет права обычного пользователя и дополнительный доступ к административному HTTP-эндпоинту `/users`. |

### 3.2. Правила назначения ролей

3.2.1. При первом обращении к боту пользователь должен сохраняться в системе с ролью `User`, если его Telegram ID не указан как административный.

3.2.2. Роль `Admin` должна назначаться пользователям, Telegram ID которых указаны в конфигурации приложения.

3.2.3. Список администраторов должен задаваться через переменную окружения, например `ADMIN_TELEGRAM_IDS`.

3.2.4. В проекте не требуется реализовывать отдельную команду для изменения роли пользователя.

### 3.3. Матрица прав доступа

| Действие | Владелец романа | Соавтор | Обычный пользователь без доступа | Admin |
|---|---:|---:|---:|---:|
| Создать роман | Да | Да | Да | Да |
| Просмотреть доступный роман | Да | Да | Нет | Только если является автором романа |
| Удалить роман | Да | Нет | Нет | Только если является владельцем романа |
| Добавить главу | Да | Да | Нет | Только если является автором романа |
| Изменить главу | Да | Да | Нет | Только если является автором романа |
| Удалить главу | Да | Да | Нет | Только если является автором романа |
| Добавить соавтора | Да | Нет | Нет | Только если является владельцем романа |
| Удалить соавтора | Да | Нет | Нет | Только если является владельцем романа |
| Просмотреть список пользователей через HTTP | Нет | Нет | Нет | Да |

Примечание: роль `Admin` не дает автоматического доступа ко всем романам через Telegram-бота. Это упрощает реализацию и делает правила доступа понятнее.

## 4. Требования к составу проекта

### 4.1. Репозиторий

4.1.1. Весь исходный код проекта должен храниться в одном репозитории GitHub.

4.1.2. Репозиторий должен содержать исходный код приложения, файл сборки, Dockerfile, docker-compose.yml, README.md и документы по требованиям и архитектуре.

4.1.3. История коммитов должна показывать вклад всех участников команды. Не допускается загружать весь проект одним финальным коммитом.

4.1.4. В репозиторий не должны попадать реальные токены Telegram API, ключи LLM API и административные токены.

4.1.5. Для примера конфигурации должен быть добавлен файл вида `.env.example` или раздел в README.md с перечнем переменных окружения.

### 4.2. README.md

4.2.1. В корне репозитория должен быть файл `README.md`.

4.2.2. README.md должен содержать:

1. название проекта;
2. краткое описание проекта;
3. состав команды;
4. список использованных технологий;
5. описание структуры проекта;
6. инструкцию по локальному запуску;
7. инструкцию по сборке fat JAR;
8. инструкцию по запуску через Docker;
9. инструкцию по запуску через Docker Compose;
10. список переменных окружения;
11. список команд Telegram-бота;
12. примеры взаимодействия с ботом;
13. описание HTTP-эндпоинтов;
14. ссылку или username Telegram-бота для проверки.

4.2.3. Инструкция запуска должна быть написана так, чтобы другой студент мог запустить проект без устных пояснений авторов.

### 4.3. Артефакты проекта

4.3.1. Основным исполняемым артефактом должен быть fat JAR.

4.3.2. Приложение должно запускаться из fat JAR командой вида:

```bash
java -jar app.jar
```

4.3.3. Проект должен содержать `Dockerfile` для сборки Docker-образа приложения.

4.3.4. Проект должен содержать `docker-compose.yml` для локального запуска приложения.

4.3.5. Docker-образ приложения должен быть опубликован в Docker Hub. Ссылка на образ указывается при сдаче проекта.

## 5. Требования к технологиям

### 5.1. Язык программирования

5.1.1. Для реализации проекта должен использоваться Java 25.

5.1.2. В коде должны использоваться современные возможности выбранного языка. Не допускается стиль программирования, характерный для устаревших версий Java.

### 5.2. Сборка проекта

5.2.1. Для сборки проекта должен использоваться Gradle.

5.2.2. Файл сборки должен содержать все необходимые зависимости приложения.

5.2.3. Сборка проекта должна создавать исполняемый fat JAR.

### 5.3. Framework

5.3.1. Для организации приложения и внедрения зависимостей должен использоваться Spring 7.

5.3.2. Использование Spring Boot запрещено.

5.3.3. Конфигурация Spring должна быть выполнена вручную: через Java-конфигурацию, XML-конфигурацию или их сочетание.

5.3.4. В проекте должны быть выделены компоненты для Telegram-бота, бизнес-логики, хранения данных, работы с LLM API и HTTP-интерфейса.

### 5.4. Telegram API

5.4.1. Приложение должно интегрироваться с Telegram API.

5.4.2. Бот должен принимать текстовые команды от пользователей.

5.4.3. Бот должен отправлять пользователю текстовые ответы.

5.4.4. Допускается использовать long polling. Реализация webhook не является обязательной.

### 5.5. LLM API

5.5.1. Приложение должно интегрироваться с одним LLM API.

5.5.2. В качестве LLM API допускается использовать OpenAI, Anthropic, GigaChat или другой доступный сервис.

5.5.3. Ключ LLM API должен передаваться через переменную окружения или конфигурационный файл, не публикуемый в репозитории.

5.5.4. В случае недоступности LLM API приложение не должно завершаться аварийно.

### 5.6. Очередь сообщений

5.6.1. Очередь сообщений реализует минимальную интеграцию с RabbitMQ только для передачи заданий на LLM-обработку.

## 6. Функциональные требования

### 6.1. Регистрация пользователя

6.1.1. При первом обращении пользователя к Telegram-боту система должна сохранять пользователя по его Telegram ID.

6.1.2. При регистрации должны сохраняться следующие данные:

1. Telegram ID;
2. username, если он доступен;
3. имя пользователя, если оно доступно;
4. роль пользователя;
5. дата и время регистрации.

6.1.3. Повторное выполнение команды `/start` не должно создавать дубликат пользователя.

6.1.4. Если пользователь уже зарегистрирован, команда `/start` должна выводить приветствие и список основных команд.

### 6.2. Авторизация

6.2.1. В Telegram-боте пользователь считается авторизованным, если его Telegram ID получен из сообщения Telegram.

6.2.2. Для HTTP-интерфейса должна быть предусмотрена простая авторизация по административному токену.

6.2.3. Административный токен должен передаваться в HTTP-заголовке. Рекомендуемое имя заголовка: `X-Admin-Token`.

6.2.4. Если административный токен отсутствует или неверен, доступ к защищенному HTTP-эндпоинту должен быть запрещен.

### 6.3. Общие команды Telegram-бота

6.3.1. Telegram-бот должен поддерживать команду `/start`.

Результат выполнения команды:

1. регистрация пользователя, если он еще не зарегистрирован;
2. вывод приветствия;
3. вывод краткого списка основных команд.

6.3.2. Telegram-бот должен поддерживать команду `/help`.

Результат выполнения команды — вывод справки по доступным командам.

6.3.3. Команда `/help` должна содержать не только названия команд, но и краткие примеры их использования.

6.3.4. При вводе неизвестной команды бот должен сообщить, что команда не распознана, и предложить использовать `/help`.

6.3.5. Команды должны быть реализованы в текстовом формате. Допускается использовать разделитель `|` для передачи нескольких текстовых аргументов.

### 6.4. Управление романами

6.4.1. Пользователь должен иметь возможность создать роман.

Данные для создания романа:

1. название;
2. описание;
3. жанр.

Пример команды:

```text
/new_novel Звезды над городом | История о будущем мегаполисе | фантастика
```

6.4.2. После создания романа пользователь, создавший роман, становится его владельцем.

6.4.3. При создании романа должны сохраняться:

1. идентификатор романа;
2. название;
3. описание;
4. жанр;
5. Telegram ID владельца;
6. дата создания;
7. дата последнего изменения.

6.4.4. Пользователь должен иметь возможность получить список романов, к которым он имеет доступ.

В список должны входить:

1. романы, созданные пользователем;
2. романы, где пользователь является соавтором.

6.4.5. В списке романов должны отображаться идентификатор, название, жанр и роль пользователя в романе.

6.4.6. Пользователь должен иметь возможность просмотреть краткую информацию о романе.

Краткая информация должна содержать:

1. идентификатор романа;
2. название;
3. описание;
4. жанр;
5. владельца;
6. список авторов;
7. количество глав;
8. дату создания.

6.4.7. Владелец романа должен иметь возможность удалить роман.

6.4.8. При удалении романа должны удаляться его главы и история изменений глав.

6.4.9. Соавтор не должен иметь возможность удалить роман, если он не является владельцем.

6.4.10. Пользователь без доступа к роману не должен видеть сведения о нем.

### 6.5. Совместная работа

6.5.1. Владелец романа должен иметь возможность добавить соавтора к роману по Telegram ID пользователя.

Пример команды:

```text
/invite_author 3 123456789
```

где `3` — идентификатор романа, `123456789` — Telegram ID пользователя.

6.5.2. Добавлять можно только уже зарегистрированного пользователя, то есть пользователя, который хотя бы один раз запускал бота командой `/start`.

6.5.3. Если пользователь с указанным Telegram ID не найден, бот должен вывести сообщение об ошибке.

6.5.4. Нельзя повторно добавить пользователя, который уже является автором романа.

6.5.5. Владелец романа должен иметь возможность удалить соавтора из романа.

6.5.6. Владелец романа не может быть удален из списка авторов.

6.5.7. Пользователь должен иметь возможность просмотреть список авторов конкретного романа.

6.5.8. В списке авторов должны отображаться Telegram ID, username при наличии и роль в рамках романа: владелец или соавтор.

### 6.6. Работа с главами

6.6.1. Автор романа должен иметь возможность добавить главу к роману.

Данные главы:

1. название главы;
2. текст главы.

Пример команды:

```text
/add_chapter 3 | Глава 1. Начало | Вечером город светился холодным синим светом...
```

6.6.2. При добавлении главы должны сохраняться:

1. идентификатор главы;
2. идентификатор романа;
3. название главы;
4. текст главы;
5. номер главы или порядок добавления;
6. автор последнего изменения;
7. дата создания;
8. дата последнего изменения.

6.6.3. Автор романа должен иметь возможность получить список глав романа.

6.6.4. В списке глав должны отображаться идентификатор главы, номер главы и название.

6.6.5. Автор романа должен иметь возможность просмотреть текст конкретной главы.

6.6.6. Автор романа должен иметь возможность просмотреть полный текст романа.

6.6.7. Полный текст романа должен формироваться из глав в порядке их добавления.

6.6.8. Автор романа должен иметь возможность изменить название и текст главы.

6.6.9. При изменении главы предыдущая версия должна сохраняться в истории изменений.

6.6.10. Автор романа должен иметь возможность удалить главу.

6.6.11. Автор романа должен иметь возможность просмотреть историю изменений главы.

6.6.12. История изменений может хранить только последние 5 версий главы. Этого достаточно для демонстрации функции и не усложняет реализацию.

6.6.13. Пользователь без доступа к роману не должен иметь возможности просматривать, изменять или удалять его главы.

### 6.7. История изменений главы

6.7.1. При каждом изменении главы система должна сохранять предыдущую версию текста.

6.7.2. Для каждой записи истории должны сохраняться:

1. идентификатор записи истории;
2. идентификатор главы;
3. старое название главы;
4. старый текст главы;
5. Telegram ID пользователя, выполнившего изменение;
6. дата и время изменения.

6.7.3. Команда просмотра истории должна показывать список версий без вывода полного текста всех версий, чтобы ответ Telegram-бота не был слишком длинным.

6.7.4. Для просмотра конкретной версии допускается реализовать отдельную команду. Если отдельная команда не реализуется, достаточно показывать краткие фрагменты старых версий.

### 6.8. LLM-функции

6.8.1. Автор романа должен иметь возможность запросить у LLM продолжение текста главы.

Пример команды:

```text
/continue_chapter 3 2
```

где `3` — идентификатор романа, `2` — идентификатор главы.

6.8.2. Результат выполнения команды `/continue_chapter` — текстовое предложение продолжения главы.

6.8.3. Сгенерированный текст не должен автоматически сохраняться в романе. Пользователь при необходимости может самостоятельно добавить его через команду изменения или добавления главы.

6.8.4. Автор романа должен иметь возможность запросить совет по развитию сюжета или персонажей.

Пример команды:

```text
/advice 3 Как лучше развить конфликт между героями?
```

6.8.5. Результат выполнения команды `/advice` — краткий ответ LLM с рекомендациями.

6.8.6. Автор романа должен иметь возможность создать черновик главы по пользовательскому запросу.

Пример команды:

```text
/draft 3 Напиши сцену знакомства главного героя с антагонистом
```

6.8.7. Результат выполнения команды `/draft` — черновик текста. Черновик не должен автоматически добавляться в роман.

6.8.8. При формировании запроса к LLM приложение должно передавать краткий контекст романа: название, жанр, описание и при необходимости фрагмент текста главы.

6.8.9. Чтобы запросы к LLM не были слишком большими, приложение может передавать только последние 1000--2000 символов текста главы.

6.8.10. При ошибке LLM API бот должен вывести понятное сообщение, например:

```text
Не удалось получить ответ от языковой модели. Попробуйте позже.
```

6.8.11. Если ключ LLM API не задан, бот должен сообщать, что LLM-функции временно недоступны.

### 6.9. HTTP-интерфейс

6.9.1. Приложение должно иметь HTTP-эндпоинт `/healthcheck`, доступный без авторизации.

Ответ `/healthcheck` должен содержать:

1. статус приложения;
2. название проекта;
3. список авторов проекта;
4. текущую дату и время сервера.

Пример ответа:

```json
{
  "status": "OK",
  "project": "Collaborative Novel Writing Bot",
  "authors": ["Student 1", "Student 2"],
  "time": "2026-04-16T12:00:00"
}
```

6.9.2. Приложение должно иметь HTTP-эндпоинт `/users`, доступный только администратору.

Ответ `/users` должен содержать список зарегистрированных пользователей:

1. Telegram ID;
2. username, если он доступен;
3. роль пользователя;
4. дата регистрации.

6.9.3. При обращении к `/users` без административного токена сервер должен возвращать ошибку доступа.

6.9.4. Для `/healthcheck` рекомендуется использовать HTTP-метод `GET`.

6.9.5. Для `/users` рекомендуется использовать HTTP-метод `GET`.

6.9.6. Не требуется реализовывать полный REST API для романов и глав. Работа с романами и главами выполняется через Telegram-бота.

## 7. Требования к формату команд Telegram-бота

7.1.2. Аргументы команды должны передаваться после имени команды.

7.1.3. Для команд с несколькими длинными текстовыми аргументами должен использоваться разделитель `|`.

7.1.4. Если пользователь передал недостаточно аргументов, бот должен вывести пример корректного использования команды.

7.1.5. Если пользователь передал идентификатор романа или главы в неверном формате, бот должен вывести сообщение об ошибке.

| `/draft` | генерация черновика главы через LLM | `/draft 3 Напиши сцену знакомства` |

### 7.3. Требования к ответам бота

7.3.1. Все ответы бота должны быть на русском языке.

7.3.2. Ответы должны быть понятными без чтения исходного кода.

7.3.3. После успешного выполнения команды бот должен сообщать, что действие выполнено.

7.3.4. При ошибке бот должен сообщать причину ошибки в краткой форме.

7.3.5. Если текст главы или полный текст романа слишком длинный для одного сообщения Telegram, допускается выводить только начало текста и сообщение о том, что текст был сокращен.

### 8.3. Минимальная структура данных

4. `oldText`;

## 9. Нефункциональные требования

### 9.1. Производительность

9.1.1. Бот должен обрабатывать обычные команды пользователя без заметной задержки, если команда не обращается к LLM API.

9.1.2. Для команд, обращающихся к LLM API, допускается более длительное ожидание ответа.

9.1.3. Система должна быть рассчитана на учебную нагрузку: до 20 зарегистрированных пользователей, до 50 романов и до 300 глав суммарно.

9.1.4. Не требуется реализовывать горизонтальное масштабирование приложения.

### 9.2. Надежность

9.2.1. При некорректной команде бот должен выводить сообщение с пояснением ошибки и примером правильного формата.

9.2.2. При отсутствии прав доступа к роману или главе бот должен сообщать пользователю, что у него нет доступа.

9.2.3. Ошибки Telegram API, LLM API и хранения данных должны обрабатываться без аварийного завершения приложения.

9.2.4. При запуске приложения без обязательных переменных окружения приложение должно выводить понятное сообщение об ошибке.

9.2.5. Если данные не удалось сохранить, бот должен сообщить пользователю, что действие не выполнено.

### 9.3. Безопасность

9.3.1. Токены Telegram API, LLM API и административный HTTP-токен не должны публиковаться в GitHub.

9.3.2. Пользователь не должен иметь доступ к романам, где он не является владельцем или соавтором.

9.3.3. Только владелец романа может удалять роман и управлять списком соавторов.

9.3.4. HTTP-эндпоинт `/users` должен быть доступен только администратору.

9.3.5. В сообщениях об ошибках не должны выводиться токены, ключи API и другие служебные секреты.

### 9.4. Удобство использования

9.4.1. Все сообщения Telegram-бота должны быть написаны на русском языке.

9.4.2. Ответы бота должны быть достаточно короткими, чтобы их было удобно читать в Telegram.

9.4.3. Команда `/help` должна содержать список всех основных команд с примерами.

9.4.4. Идентификаторы романов и глав должны показываться пользователю в списках, чтобы он мог использовать их в командах.

9.4.5. Для длинных ответов допускается сокращать текст и указывать, что ответ был сокращен.

### 9.5. Сопровождаемость

9.5.1. Код должен быть разделен на логические части.

9.5.2. Основная бизнес-логика не должна находиться непосредственно в классе Telegram-бота.

9.5.3. Классы для работы с Telegram, LLM API, хранением данных и HTTP-запросами должны быть отделены друг от друга.

9.5.4. Названия классов, методов и переменных должны отражать их назначение.

Экспорт в PDF, сложные напоминания и полноценная проверка противоречий не рекомендуются для обязательной реализации, так как они могут занять слишком много времени.

## 17. Критерии приемки проекта

Проект считается выполненным, если:

1. приложение собирается в fat JAR;
2. приложение запускается локально;
3. приложение запускается через Docker Compose;
4. Telegram-бот отвечает на команды `/start` и `/help`;
5. пользователь может создать роман;
6. пользователь может добавить главу к роману;
7. пользователь может просмотреть список своих романов;
8. пользователь может просмотреть текст главы;
9. пользователь может просмотреть полный текст романа или его сокращенную версию;
10. владелец романа может добавить соавтора;
11. владелец романа может удалить соавтора;
12. пользователь без доступа не может просматривать и изменять чужой роман;
13. при изменении главы сохраняется история изменений;
14. хотя бы две команды обращаются к LLM API и возвращают ответ пользователю;
15. HTTP-эндпоинт `/healthcheck` доступен без авторизации;
16. HTTP-эндпоинт `/users` доступен только администратору;
17. данные пользователей, романов и глав сохраняются после перезапуска приложения;
18. в GitHub есть README.md с инструкциями;
19. Docker-образ опубликован в Docker Hub;
20. токены и ключи не опубликованы в репозитории;
21. в проекте есть не менее пяти простых тестов;
22. в документации указаны требования, архитектура и порядок запуска проекта.
